# x402 — AI-agent payments

Eddy meters its JSON API for AI agents using the [x402](https://x402.org)
payment protocol (**V2**). Human visitors and browsers are never affected —
only automated AI agents (detected by `User-Agent`) are asked to pay, in USDC,
per request. Content pages (`/rivers`, `/blog`, `/gauges`, …) stay free for
crawlers so Eddy remains groundable and indexable.

The whole layer is **dormant until a payout address is configured**. With no
wallet set, every route serves normally and `/.well-known/x402` returns 503.

## Architecture

Built on `@x402/core@2.18` (the framework-agnostic core) rather than the
official `@x402/next` adapter, because `@x402/next@2` requires Next.js ≥ 16 and
Eddy runs on Next 14. All payment verification and settlement happens inside
`@x402/core`; Eddy only supplies a thin Next-14 request/response adapter.

| File | Responsibility |
| --- | --- |
| `src/lib/x402/config.ts` | Networks, payout addresses, single-source price table, feature flags, facilitator selection |
| `src/lib/x402/engine.ts` | Next-14 `HTTPAdapter` + verify → run handler → settle-after-success orchestration |
| `src/lib/x402-config.ts` | Public entry point (`withX402Route`, `isAiAgent`, discovery constants) |
| `src/app/.well-known/x402/route.ts` | V2 discovery manifest, generated from the price table |

Enforcement is **per-route** (there is no x402 logic in `middleware.ts`). Each
gated route wraps its handler with `withX402Route(handler, routeKey)`, where
`routeKey` indexes the single-source `X402_ROUTES` price map — so the prices
enforced on routes and the prices advertised in the manifest can never drift.

## How a paid request works

1. A browser or non-agent request → passes straight through, free.
2. An AI agent with **no** payment → HTTP **402** with a `PAYMENT-REQUIRED`
   header carrying the `accepts` list (scheme, network, USDC amount, `payTo`).
3. An AI agent **with** a valid payment → the handler runs; the payment is
   **settled only after** a successful (`<400`) response, and a
   `PAYMENT-RESPONSE` header is attached. Failed handlers are never charged.

## Environment variables

| Variable | Required | Purpose |
| --- | --- | --- |
| `X402_WALLET_ADDRESS` | to enable | EVM (Base) payout address. Setting this activates x402. |
| `X402_SOLANA_ADDRESS` | optional | Solana payout address (base58). Adds a Solana accept option alongside Base. |
| `CDP_API_KEY_ID` / `CDP_API_KEY_SECRET` | recommended | Coinbase CDP facilitator credentials. Enables mainnet settlement **and** automatic [Bazaar](https://docs.cdp.coinbase.com/x402/bazaar) discovery indexing. |
| `X402_FACILITATOR_URL` | optional | Explicit facilitator URL, used only when CDP creds are absent. Defaults to the public `https://x402.org/facilitator` (testnet-only). |
| `X402_TESTNET` | optional | `true` → price on Base Sepolia / Solana devnet (pair with the public facilitator to test without real funds). Default: mainnet. |
| `X402_SIWX_ENABLED` | optional | Wallet-based sessions (Sign-In-With-X / CAIP-122). See caveat below. |
| `X402_BAZAAR_ENABLED` | optional | Register the server-side Bazaar extension (discovery also works via the CDP facilitator). |
| `X402_DYNAMIC_PAYTO` | optional | Use a server-side dynamic `payTo` resolver (for per-request revenue routing). The client never chooses the recipient. |

## Facilitator & discovery

The facilitator verifies and settles payments. Eddy picks one automatically:

1. **Coinbase CDP** when `CDP_API_KEY_ID`/`CDP_API_KEY_SECRET` are set — settles
   Base mainnet and **auto-indexes Eddy's paid endpoints in the x402 Bazaar**
   the first time a payment settles (no separate registration).
2. Otherwise the explicit `X402_FACILITATOR_URL`.
3. Otherwise the public `https://x402.org/facilitator` (testnet networks only).

> For production mainnet, use the Coinbase CDP facilitator. The public
> facilitator only supports testnets (Base Sepolia, Solana devnet).

## Multi-chain

Set `X402_WALLET_ADDRESS` for Base and, optionally, `X402_SOLANA_ADDRESS` for
Solana. When both are set, each 402 offers **both** chains in `accepts` and the
paying agent chooses.

## Wallet sessions (SIWx) — caveat

`X402_SIWX_ENABLED` lets an agent that already paid for a resource re-access it
by presenting a Sign-In-With-X proof instead of paying again. The default
storage is **in-memory**, which does **not** persist across serverless
instances/cold starts on Vercel. Before enabling in production, supply a
persistent `SIWxStorage` (e.g. Supabase- or Redis-backed) in
`src/lib/x402/engine.ts` (`getSiwxStorage`).

## Local testing

```bash
# Testnet mode + a throwaway payout wallet → uses the public facilitator.
X402_TESTNET=true \
X402_WALLET_ADDRESS=0x0000000000000000000000000000000000000001 \
npm run dev

# Manifest (V2):
curl localhost:3000/.well-known/x402

# Unpaid AI agent → 402 with a PAYMENT-REQUIRED header:
curl -i -H 'User-Agent: ClaudeBot/1.0' localhost:3000/api/vessel-types

# Browser → free passthrough (never 402):
curl -i -H 'User-Agent: Mozilla/5.0' localhost:3000/api/vessel-types
```

## Adding a new paid route

1. Add the path + price to `X402_ROUTES` in `src/lib/x402/config.ts`.
2. Wrap the handler export: `export const GET = withX402Route(_GET, '/api/your-route');`
   (dynamic routes pass their context type, e.g.
   `withX402Route<{ params: Promise<{ slug: string }> }>(_GET, '/api/x/:slug')`).

The manifest updates itself from the price table — no second edit needed.
